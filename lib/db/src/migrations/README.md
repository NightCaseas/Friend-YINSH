# Миграции базы данных

## Обзор

Эта директория содержит SQL миграции для базы данных Friends YINSH. Миграции позволяют безопасно изменять схему базы данных с сохранением данных.

## Структура

Файлы миграций именуются по шаблону:
```
{номер}_описание.sql
```

Где:
- `{номер}` - трехзначный номер миграции в порядке применения
- `описание` - краткое описание изменений на английском языке

## Существующие миграции

### 001_add_users_tables.sql
**Добавление таблиц для системы пользователей, аутентификации и друзей**

**Изменения:**
1. Создание таблицы `users` для хранения основной информации о пользователях
2. Создание таблицы `sessions` для хранения сессий и refresh токенов
3. Создание таблицы `friends` для управления дружескими отношениями
4. Создание таблицы `game_stats` для хранения игровой статистики
5. Добавление полей `user1_id`, `user2_id`, `created_by` в таблицу `games`
6. Создание индексов для оптимизации запросов
7. Создание триггеров для автоматического обновления `updated_at`
8. Создание функции и триггера для автоматического обновления статистики пользователей

## Применение миграций

### Автоматическое применение (рекомендуется)
```bash
# Из корня проекта
npm run db:migrate

# Или напрямую через drizzle-kit
npx drizzle-kit migrate
```

### Ручное применение
```bash
# Подключение к базе данных
psql -U postgres -d friend_yinsh

# Применение конкретной миграции
\i src/migrations/001_add_users_tables.sql
```

## Создание новых миграций

### 1. С использованием Drizzle Kit
```bash
# Генерация миграции на основе изменений схемы
npx drizzle-kit generate:pg

# Проверка сгенерированной миграции
cat drizzle/migrations/0001_миграция.sql
```

### 2. Ручное создание
1. Создайте файл в формате `XXX_description.sql`
2. Напишите SQL команды для изменения схемы
3. Добавьте комментарии для каждого изменения
4. Убедитесь, что миграция идемпотентна (используйте `IF NOT EXISTS`, `IF EXISTS`)

## Принципы

### 1. Идемпотентность
Все миграции должны быть идемпотентными - их можно безопасно применять несколько раз без побочных эффектов.

**Правильно:**
```sql
CREATE TABLE IF NOT EXISTS users (...);
ALTER TABLE games ADD COLUMN IF NOT EXISTS user1_id UUID;
```

**Неправильно:**
```sql
CREATE TABLE users (...);
ALTER TABLE games ADD COLUMN user1_id UUID;
```

### 2. Обратная совместимость
Миграции должны сохранять обратную совместимость:
- Не удаляйте колонки без деприкационного периода
- Не изменяйте типы данных без преобразования существующих данных
- Добавляйте новые колонки со значениями по умолчанию

### 3. Безопасность данных
- Всегда делайте резервную копию перед применением миграций
- Тестируйте миграции на тестовой базе
- Используйте транзакции для критических изменений

## Восстановление (Rollback)

### Для production
1. Создайте новую миграцию для отката изменений
2. Примените миграцию восстановления

### Для разработки
```bash
# Восстановление до предыдущего состояния
npx drizzle-kit drop

# Пересоздание базы данных
npx drizzle-kit push
```

### Статистика базы данных
```sql
-- Количество пользователей
SELECT COUNT(*) FROM users;

-- Количество активных сессий
SELECT COUNT(*) FROM sessions WHERE expires_at > NOW();

-- Статистика дружеских отношений
SELECT status, COUNT(*) FROM friends GROUP BY status;
```

## Устранение неполадок

### 1. Ошибка при применении миграции
```bash
# Проверьте SQL синтаксис
psql -U postgres -d friend_yinsh -f src/migrations/001_add_users_tables.sql --echo-all

# Проверьте существующие объекты
psql -U postgres -d friend_yinsh -c "\d users"
```

### 2. Конфликт миграций
Если возникает конфликт между миграциями:
1. Проверьте порядок применения миграций
2. Убедитесь, что номера миграций уникальны
3. При необходимости переименуйте файл миграции

### 3. Проблемы с производительностью
Для больших таблиц:
1. Создавайте индексы после вставки данных
2. Используйте `CONCURRENTLY` для создания индексов на production
3. Разбивайте большие миграции на несколько частей

## Лучшие практики

- Всегда пишите миграции перед изменением кода
- Тестируйте миграции локально перед коммитом

### 3. Развертывание
- Применяйте миграции во время низкой нагрузки
- Мониторьте производительность после применения
- Имейте план отката на случай проблем
